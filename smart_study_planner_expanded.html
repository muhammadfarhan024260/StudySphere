Smart Study Planner with Analytics — expanded

Tracks student study sessions, goals, and weak areas. Generates daily/weekly analytics reports. Admin manages subjects and recommendations.

Education domain 9 tables Admin + Student roles
Original had 4 tables and weak pattern fit. This version expands to 9 tables, adds 5 natural DBMS feature hooks, and gives every pattern a genuine use case — not a forced one.

Database schema — 9 tables

Student

Core user entity

StudentID PK Name Email PasswordHash CreatedAt Role = 'student'
Admin

Manages subjects & recommendations

AdminID PK Name Email Role = 'admin'
Subject

Subjects tracked in the system

SubjectID PK AdminID FK SubjectName Category TargetHours
StudyLog

Core fact table — every session

LogID PK StudentID FK SubjectID FK HoursStudied ProductivityScore SessionDate Notes
Goal

Weekly/monthly targets per subject

GoalID PK StudentID FK SubjectID FK TargetHours Deadline Status
WeakArea

Auto-flagged by trigger

WeakAreaID PK StudentID FK SubjectID FK AvgScore FlaggedOn Resolved
Notification

Decorated at runtime (Decorator)

NotifID PK StudentID FK Type Message SentAt IsRead
ReportTemplate

Adapted by Adapter pattern

TemplateID PK AdminID FK Format TemplateName CreatedAt
Recommendation

Admin-created suggestions for weak areas

RecID PK AdminID FK SubjectID FK ResourceLink MinScoreThreshold
PK = Primary Key FK = Foreign Key Key analytics column

DBMS lab coverage — all 13 labs mapped

Lab 1 — SELECT Basic queries List all study logs for a student; filter by date range
Lab 2 — ORDER/GROUP GROUP BY + ORDER BY Group logs by subject, order by total hours DESC
Lab 3 — Aggregates SUM, AVG, COUNT, MAX AVG(ProductivityScore) per subject; SUM(HoursStudied) weekly
Lab 4 — Joins INNER / LEFT JOINs StudyLog → Subject → Recommendation → Student (4-table join)
Lab 5 — Set ops UNION / INTERSECT Subjects studied this week UNION subjects with active goals
Lab 6 — Subqueries Nested SELECT Students whose AVG score is below the overall system average
Lab 7 — DDL CREATE / ALTER All 9 tables with PKs, FKs, constraints, NOT NULL
Lab 8 — DML INSERT / UPDATE / DELETE Log a new study session; mark goal complete; delete old logs
Lab 9 — Views CREATE VIEW vw_WeeklyReport — pre-joined summary view per student
Lab 10 — Stored Procs CREATE PROCEDURE sp_GetWeakSubjects(@StudentID) — returns subjects below threshold
Lab 11 — Triggers AFTER INSERT trigger After INSERT on StudyLog — if AVG score < 50, auto-insert into WeakArea
Lab 12 — ROLLUP ROLLUP / CUBE ROLLUP on (StudentID, SubjectID, Month) for analytics dashboard
Lab 13 — Connectivity ASP.NET + SQL CRUD Full C# web app — log sessions, view reports, manage goals

SDA pattern mapping — natural fits only

Pattern Where it lives What it does
Singleton DatabaseConnection class One shared SQL connection instance across all data access classes — prevents multiple open connections
Decorator Notification system Base notification gets decorated at runtime — EmailDecorator, SMSDecorator, PushDecorator wrap each other to add delivery channels without changing core logic
Adapter Report export engine IReportGenerator interface — PDFReportAdapter and CSVReportAdapter adapt the same analytics data to different output formats. Admin picks format; system adapts.
MVC Overall architecture Model = data classes (Student, StudyLog). View = ASP.NET pages. Controller = business logic (calculate weak areas, generate reports)
3-Tier / Façade System layers Presentation → Business Logic → Data Access. StudyPlannerFacade exposes simple methods (LogSession, GetReport) hiding complex joins and procedures behind one interface
